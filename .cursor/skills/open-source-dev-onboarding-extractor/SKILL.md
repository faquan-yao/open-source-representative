# 开源项目开发者概览提取

从任意开源仓库提取**开发者向项目概览**。让读者在 **10～15 分钟**内明白：项目做什么、怎么搭、主链路怎么走、从哪改代码、如何跑起来。

**适度精炼，而非极简或冗长：**

- **保留：** 问题域、组件表、主链路图+表（含关键路径）、代码导航、关键配置、运行命令、踩坑提示、延伸阅读
- **压缩：** 配置只列 3～6 项、代码导航 6～8 行、延伸 5～7 条、测试/贡献各 2～3 句
- **不写进终稿：** 完整 CI 矩阵、10 文件清单、file:function 逐步表、与架构图/主链路图重复的第三张及以上图示

本 Skill **不**用于终端用户营销文案。

## 路径约定

**{项目根}** = 被分析开源仓库的根目录。阶段 0 须确认。

**{产出目录}** = 用户指定目录；未指定则 `{项目根}/项目汇总/`。须确认并创建。

## 产出路径（必遵）

| 文件 | 路径 |
|------|------|
| 草稿 | `{产出目录}/dev-onboarding-scratch.md` |
| 最终报告 | `{产出目录}/developer-onboarding-report.md` |

## 操作原则

1. **先读后写** — 结论可核对；未知标 `未验证` 或 `未找到`。详细依据进草稿。
2. **关键信息不丢** — 主链路、入口路径、运行命令、边界、踩坑必须出现在终稿。
3. **通顺中文** — 合成前阅读 [references/writing-style.md](references/writing-style.md)；禁止「是一个 Python CLI 与库」类硬拼句式；摘要须朗读通顺。
4. **两张图** — 终稿须含 **整体架构图**（6～10 节点）与 **主链路图**（4～6 步）；主链路另附表格列出每步关键 `path`，图与表互补、勿重复第三张图。
5. **中文产出** — 简体中文叙述；项目名、路径、命令、符号名保留原文。

## 工作流

```
提取进度：
- [ ] 阶段 0：侦察
- [ ] 阶段 1：草稿收集
- [ ] 阶段 2：现场验证（可选）
- [ ] 阶段 3：合成报告 + 中文润色
```

### 阶段 0：侦察（15～25 分钟）

1. 阅读：`README*`、`ARCHITECTURE*`、`CONTRIBUTING*`、`docs/` 索引、包清单。
2. 识别：语言、入口、组件、外部依赖、主链路触发点。
3. 写入 `{产出目录}/dev-onboarding-scratch.md`。
4. 停止条件：能用**一段通顺中文**说清项目做什么、给谁用、怎么跑。

见 [references/extraction-checklist.md](references/extraction-checklist.md)。

### 阶段 1：草稿收集

在 scratch 中记录（供合成提炼，勿原样粘贴）：

| 维度 | 终稿对应节 |
|------|------------|
| 问题域、角色、场景、边界 | 问题域与边界 |
| 组件、外部依赖 | 整体架构图 + 主要组件表 |
| 主链路 4～6 步 + 路径 | 主执行链路（图 + 表） |
| 能力 → 入口 | 代码导航 |
| 关键配置、运行命令 | 配置与运行 |
| 测试命令、贡献流程 | 配置与运行（摘要） |
| 地雷、文档漂移 | 注意事项 |
| 重要文档 | 延伸阅读 |

### 阶段 2：现场验证（可行时）

1. 文档中的安装/构建/运行命令（或标 `未验证`）。
2. 可选：一条与主链路相关的 smoke 测试。
3. 结果写入终稿「验证记录」一行。

### 阶段 3：合成报告

1. 按 [templates/developer-onboarding-report.md](templates/developer-onboarding-report.md) 输出。
2. 填满各节；占位符替换为仓库具体内容。
3. **中文润色（必做）：** 对照 [references/writing-style.md](references/writing-style.md) 改写摘要与各节首句；读出声检查通顺度。
4. 交付 `{产出目录}/developer-onboarding-report.md`。

### 终稿体量（参考）

| 约束 | 标准 |
|------|------|
| 总长度 | 约 **150～250 行** |
| 架构图 | **1 张**，6～10 节点 |
| 主链路图 | **1 张**，4～6 步，与架构图分工（架构看组件，主链路看调用顺序） |
| 主链路表 | **4～6 步**，每步 1 个关键 `path` |
| 代码导航 | **6～8 行** |
| 关键配置 | **3～6 项** |
| 延伸阅读 | **5～7 条** |

## 产出质量门槛

| 检查项 | 通过标准 |
|--------|----------|
| **讲明白** | 读者能说出做什么、主链路、从哪改、怎么跑 |
| **关键信息** | 含边界、路径锚点、运行命令、≥2 条注意事项或延伸 |
| **通顺中文** | 无「是一个 X CLI 与库」类语病；摘要 3～5 句完整可读 |
| **有据** | 不编造命令、模块、配置键 |
| **够精炼** | 无完整 CI/lint 表、无 10 文件清单、无 directory tree |
| **图示** | 2 张 Mermaid（架构 + 主链路），语法正确、标签中文 |

## 深度模式

| 模式 | 适用 | 范围 |
|------|------|------|
| **标准** | 默认 | 上述模板 |
| **详细** | 用户明确要求 | 标准 + 附录：完整测试门禁、架构原则、公开 API 表 |

默认 **标准**。

## 反模式

- 不要删掉主链路路径、运行命令、边界等关键信息。
- 不要英文 README 直译腔（见 writing-style.md 差例）。
- 不要把 scratch 全文粘贴进终稿。
- 不要配置键全表、不要必读 10 文件、不要 dump `tree`。
