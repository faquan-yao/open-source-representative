# 中文表述规范

终稿须**通顺可读**，像工程师写给同事的项目介绍，而非英文 README 的机械翻译。

## 核心原则

1. **先中文、后术语** — 英文缩写放在中文释义之后，首次出现写全，例如「命令行工具（CLI）」。
2. **完整句，不堆标签** — 每段用完整主谓宾；禁止把技术名词用「与」「及」串成短语当句子。
3. **说清关系** — 不只罗列「CLI、库、API」，要说清它们各自承担什么、如何配合。
4. **合成后朗读** — 读出声卡顿或像翻译腔的句子，必须改写后再交付。

## 禁止写法

| 类型 | 差例 | 问题 |
|------|------|------|
| 硬拼名词 | 「X 是一个 Python CLI 与库」 | 「是一个」后接英文名词堆叠，不成句 |
| 标签堆砌 | 「开源、高性能、可扩展的 Gateway 平台」 | 只有形容词，不知道做什么 |
| 翻译腔 | 「用于实现对上游工具的选型、安装、体检」 | 「实现对…的」冗长别扭 |
| 空洞定位 | 「一站式 AI Agent 基础设施」 | 无具体场景，读者无画面 |

## 推荐写法

| 场景 | 好例 |
|------|------|
| 形态说明 | 「Agent Reach 基于 Python 开发，既提供命令行工具（CLI），也提供可嵌入的 Python 库。」 |
| 职责说明 | 「它负责帮 Agent 挑选、安装并检测上游工具是否可用；内容读取则直接调用 twitter-cli、yt-dlp 等，不在 Python 层二次封装。」 |
| 组件说明 | 「网关接收来自 Telegram 的消息，转发给后端的对话引擎处理。」 |
| 场景说明 | 「你在 Telegram 发一条消息，几秒内收到 AI 回复。」 |

## 摘要模板（开发者向）

按此顺序组织 3～5 句，每句只承载一个信息点：

1. **做什么** — 解决什么问题（动词开头或「这是一款…」）
2. **给谁** — 主要使用者角色
3. **怎么做** — 技术形态与关键组件（自然中文 + 必要术语）
4. **依赖什么** — 关键外部系统（一句）
5. **（可选）边界** — 明确不做什么

**示例（Agent Reach）：**

> Agent Reach 帮 AI Agent 打通访问 Twitter、YouTube 等平台的通路，解决各平台 API 门槛与工具选型问题。面向需要让 Agent「能上网」的开发者。项目基于 Python，提供命令行工具（CLI）和可嵌入的库；安装与体检走内置的 `doctor` 流程，实际内容读取委托给 twitter-cli、yt-dlp 等上游 CLI，而非在 Python 层重新包装。它不替代各平台官方 SDK，也不托管你的账号凭据。

## 图中与表中文字

- 节点标签：中文为主，如「命令行入口」「上游工具检测」
- 保留英文：项目名、命令、路径、协议名（HTTP、gRPC）
- 表格「职责」列：写**动词短语**，如「路由入站消息」「持久化会话状态」
